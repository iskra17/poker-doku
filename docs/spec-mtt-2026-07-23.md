# MTT(멀티테이블 토너먼트) 세부 기획서 — 2026-07-23

리서치 근거: `docs/research-mtt-2026-07-23.md`.
목표: **50명 미만 MTT를 1차 출시**하고, 이상 없으면 100 → 200 → 1000명으로 확장한다.
단순 기능 구현이 아니라 **토너먼트 운영(디렉터 콘솔·백오피스)**까지 포함해, 이후
**도쿠 클럽**(클럽 운영자가 자기 클럽의 캐시/토너를 직접 개설·운영·감독)으로 확장 가능한
구조를 처음부터 깐다.

---

## 0. 설계 원칙

1. **기존 불변식을 깨지 않는다** — 서버 권위, 핸드 사이에만 좌석 변경(pendingRemoval 패턴),
   팟 회계 rebuildPots, 좌석 배열 seatIndex 오름차순, revealed 계약. MTT는 이 위에 얹는 레이어다.
2. **아레나 패턴 준수** — RoomManager를 수정하는 대신 상위 매니저(TournamentManager)가 공개 API로
   방들을 오케스트레이션한다. RoomManager/엔진 수정은 "MTT 모드 분기"가 필요한 최소 지점으로 한정.
3. **규칙은 업계 표준 채택** — TDA/PokerStars 규칙(리서치 §2)을 기본값으로. 우리가 발명하지 않는다.
4. **운영자 역할을 처음부터 추상화** — v1의 "개설자(호스트)" 권한이 곧 도쿠 클럽의 "클럽 운영자"
   권한으로 승격된다. 디렉터 API는 처음부터 `directorProfileId` 기준으로 설계.
5. **캐릭터 정체성 유지** — 테이블 이동·파이널·탈락은 갸루게 연출 기회다(이사 컷인, 좌석 선택 연출 등).
   봇 16캐릭터가 필드를 채워 "혼자여도 50인 토너"가 되는 것이 차별화 축.

## 2. 토너먼트 수명주기 (상태머신)

```
REGISTERING ──(정원/시작조건)──> CHECKIN ──(체크인 마감)──> RUNNING
     │                              │                        │  ⇄ BREAK (자동, 스케줄)
     └──(취소/미달)──> CANCELLED <──┘                        │  ⇄ PAUSED (디렉터 수동)
                                                             ├──> FINAL_TABLE (잔존 ≤ 정원)
                                                             └──> COMPLETED (1명 생존)
```

- **REGISTERING**: 로비에 노출, 등록/취소 자유. 등록 시 wallet은 바이인+수수료 에스크로
  (기존 `reserveSngEntry` 확장), practice는 무료. 등록 상한 = maxEntrants.
- **CHECKIN** (노쇼 방지 — 리서치 §3-3 #4): 시작 시각 3분 전(설정 가능)부터 전원에게 체크인 프롬프트.
  시작 시각 도달 시 **체크인 완료자만 좌석 배정**, 미체크인 등록자는 자동 환불 후 제외.
  Poker Now의 탑승 그룹식은 과잉 — 단순형("준비 완료" 버튼 1개)으로.
  접속 중인 등록자는 자동 체크인 처리(접속 자체가 출석 증명).
- **RUNNING**: 좌석 랜덤 배치(테이블별 인원 균등), 단일 토너먼트 시계 가동. 이하 §4.
- **BREAK**: 블라인드 스케줄에 브레이크 레벨 삽입(기본: 레벨 6 종료 후 5분). 전 테이블 핸드 완료 후
  동시 정지(배리어 — H4H와 같은 메커니즘 재사용). 시계는 계속 흐르되 브레이크 레벨 동안 딜 중단.
- **PAUSED**: 디렉터 수동. 전 테이블 현재 핸드 완료 후 정지. 정지 중 블라인드 구조 수정 허용
  (Poker Now 방식 — 별도 "레벨 조정" 기능을 대체).
- **FINAL_TABLE**: 잔존 ≤ 테이블 정원이 되면 마지막 브레이크로 자연 구성. v1은 리드로 생략(온라인 관행),
  BGM 전환 + 시스템 공지 연출만. v1.5에서 GG식 좌석 선택 연출.
- **COMPLETED**: 최종 순위 확정 → wallet 페이아웃 정산 → 결과 오버레이. 방·결과는 10분 보존 후 정리
  (기존 SnG retainFinishedTournament 계약 승계).
- **CANCELLED**: 시작 전(미달/디렉터 취소) = 전액 환불. 시작 후 디렉터 취소 = **환불 규칙 사전 고지**
  (v1: 잔존자에게 "바이인 균등 반환 + 이미 확정된 상금 유지" — 규칙을 로비에 명시하는 것 자체가
  차별화. 리서치 §3-2 P1).

## 4. 아키텍처

### 4-1. 전체 구조

```
TournamentManager (신규 src/server/tournament-manager.ts)
 ├─ Map<tournamentId, TournamentInstance>
 │    ├─ config (블라인드 프리셋, 페이아웃, 정원, economyMode, directorProfileId…)
 │    ├─ status (상태머신 §2) + 단일 시계(startedAt, pauseAccum)
 │    ├─ entrants: 등록부 (profileId 귀속! — 세션 토큰 아님, 리서치 §3-3 #2)
 │    ├─ tables: Set<roomId> + 테이블별 메타(브레이크 순서, 인원)
 │    ├─ standings: 전역 순위(finishPlace, 잔존 스택 스냅샷)
 │    ├─ balancer: 이동/브레이크 판정 (v1 그리디 → v2 philios33 점수화)
 │    └─ barrier: H4H/브레이크/일시정지용 "다음 핸드 시작 허가" 게이트
 ├─ RoomManager 공개 API 사용: createRoom / disposeRoom / (신규) transferSeat
 └─ 훅 구독: handleCompletedHand → onTournamentHandComplete(roomId)
```

- **아레나 패턴 승계**: RoomManager 내부를 최소로 건드리고 위에서 오케스트레이션.
- 토너먼트 방은 `RoomConfig.gameMode: 'mtt'` + `tournamentId` 필드. 로비 방 목록에는 개별 테이블을
  노출하지 않고 **토너먼트 엔티티만 노출**(테이블은 토너먼트 상세에서만 보임).

### 4-2. 엔진 변경 (최소 침습)

리서치 §4-1의 충돌 지점 2곳을 `gameMode==='mtt'` 분기로 우회:

1. `startTournament`(상금풀=테이블 칩합) — mtt에서는 호출하지 않음. `state.tournament`는
   TournamentManager가 주입하는 **표시용 미러**(레벨/다음 레벨/전역 잔존 수/내 전역 순위 후보)로만 사용.
   `setTournamentLevel`은 재사용(매니저가 전 테이블에 브로드캐스트).
2. `checkTournamentEnd`(테이블 1명 생존=우승) — mtt에서는 비활성. 테이블 1명 생존은 밸런싱/브레이크
   신호로 매니저가 처리. `finished`는 매니저가 COMPLETED 시점에 세팅.
3. `recordFinish`/`assignFinishPlaces` — 유지하되 **전역 순위 번호를 매니저가 주입**하는 API 추가
   (예: `assignFinishPlaces(baseRank)` — baseRank = 전역 잔존 수). 동시 탈락 handStartChips 계약 그대로.
4. **BB 앤티 신규**: `postBlinds`에 ante 필드 추가(BB 좌석이 1BB 일괄 납부, 팟 회계는 totalContributed
   경유라 rebuildPots 불변식 자동 유지). `computeValidActions`/사이드팟 회귀 테스트 필수.
   ※ SnG에도 소급 적용 가능하지만 v1은 mtt 전용 플래그로 시작(기존 SnG 회귀 리스크 차단).

### 4-3. 테이블 이동 프로토콜 (최대 신규 공수)

**서버:**
- 신규 `RoomManager.transferSeat(fromRoomId, toRoomId, playerId, seatIndex)`:
  ①from 엔진 `processLeave`(pendingRemoval — **경제 정산 경로를 타지 않는 이동 전용 플래그**,
  기존 `leaveRoom`은 wallet 정산을 트리거하므로 금지) ②칩·상태 보존한 Player 객체를
  to 엔진 `addPlayer` 정렬 삽입 ③세션 `session.roomId = toRoomId` ④`socket.leave(from)` + `socket.join(to)`.
- 실행 시점: 두 테이블 모두 핸드 사이일 때만. 목적지가 핸드 중이면 다음 핸드 시작 전 대기열
  (seat-waiter 패턴 참고하되 토너먼트는 봇 양보가 없으므로 단순).
- `leaveAllSeatsExcept`는 mtt 이동 중 발동하지 않도록 가드.

**클라이언트:**
- 신규 서버→클라 이벤트 `table-move` `{tournamentId, fromRoomId, toRoomId, state}`:
  room-lost(로비행)와 달리 **로비 경유 없이** currentRoomId를 교체하고 새 스냅샷 적용.
  game-store에 핸들러 신규, `shouldApplyGameUpdate` 계약 유지.
- 연출: "테이블 이동!" 오버레이(1~2초) + 캐릭터 이사 컷인(캐릭터 정체성 — 이동 봇도 동일 연출).
  diffGameState는 roomId 전환 시 이벤트 스트림을 리셋(교차 diff 방지).
- resync: `session.roomId`가 이미 새 테이블이므로 기존 계약대로 자연 복원.

### 4-4. 광역 브로드캐스트

- Socket.io 룸 `tournament:<id>` 신규 — 참가자 전원 join. 레벨업/브레이크/버블/탈락 마일스톤/
  순위 스냅샷(스로틀 5초)/H4H 상태를 이 룸으로 방송.
- 개인 `game-update`는 기존 방 단위 유지. 토너먼트 상태는 별도 `tournament-update` 이벤트
  (스탠딩·잔존·평균 스택·다음 레벨 카운트다운) — 매 핸드마다가 아니라 **변화 시 + 5초 스로틀**로
  직렬화 비용 억제(리서치 §4-3 우려 #2).

### 4-5. 테이블 크기 가변 (사용자 요구: "테이블에 몇 명 앉을지 정할 수 있어야")

- 엔진/서버: `RoomConfig.maxPlayers` 리터럴 6 → `2~9` 정수로 일반화. 엔진 좌석 로직은 배열 기반이라
  값 확장만으로 동작(회귀 테스트로 확인). economy `===6` 하드코딩 제거(`=== maxPlayers`).
- UI 좌표: table-layout.ts 6좌표 수기 튜닝 유지 + **7~9인은 타원 파라메트릭 생성 함수** 신규
  (수기 튜닝 9좌표는 v1.5에서 QA하며 확정). v1 기본 노출은 6-max, 설정에 9-max를 실험 옵션으로.
- SnG/캐시에는 v1에서 개방하지 않음(테스트 범위 억제) — mtt 개설 폼에서만 선택.

### 4-6. 봇 정책

- practice MTT: 개설자(호스트)가 [봇 충원]으로 잔여 좌석을 봇으로 채움(SnG fill-bots 계약 승계).
  48인 토너에 휴먼 1명 + 봇 47도 허용 — "혼자여도 MTT" 차별화이자 부하 테스트 수단.
- wallet MTT: 휴먼 전용(SnG wallet과 동일 원칙 — 봇에 지갑 상금 불가).
- **AI 대사 게이팅**: 토너먼트 단위 쿨다운 추가(방별 20초에 더해 토너먼트 전체 동시 호출 상한 2~3),
  마일스톤(버블/파이널/우승)에 우선 배정. 나머지는 스크립트 폴백(리서치 §4-3 우려 #1).
- aggro-tracker/HUD 스탯은 방 단위 그대로 동작(이동 시 리셋 — v1 허용, v2에서 이월 검토).

### 4-7. 경제 (wallet)

- 등록: `reserveTournamentEntry(tournamentId, profileId, buyIn, fee)` — SnG 에스크로 패턴의
  토너먼트 단위 일반화. 등록 취소/체크인 실패/토너 취소 = 자동 환불.
- 시작: `startTournamentEscrow(tournamentId, entrantProfileIds)` — 단일 방이 아닌 **토너먼트 단위 집계**.
- 정산: COMPLETED 시 `settleTournament(tournamentId, results[{profileId, place, prize}])` —
  페이아웃 계단표 × 반올림 합계 보정. **상금은 프로필 귀속**(세션 토큰 아님).
- practice: 에스크로 없음, 상금은 표시용(기존 SnG practice와 동일).

## 5. UI/UX

### 5-1. 토너먼트 로비 (로비 신규 섹션)

- 로비에 [토너먼트] 탭/섹션: 카드에 상태 배지(등록 중 N/48 · 진행 중 · 종료), 바이인, 시작 시각,

- 상세(진행 중): **실시간 순위표**(전역 스탠딩, 내 순위 하이라이트), 잔존/평균 스택/최다 스택,
  현재 레벨·다음 레벨 카운트다운, 테이블 목록(번호·인원). 탈락자도 이 화면으로 관전(순위표만 — 테이블
  관전은 비범위).
- 개설 폼(CreateRoomModal 확장 또는 전용 모달): 이름/정원(8~48)/테이블 크기(6 기본)/블라인드 프리셋+
  레벨 길이/시작 방식(예약 시각 or 호스트 수동 시작)/economyMode/봇 충원 여부(practice).

### 5-2. 게임 내 HUD (GameRoomView 확장)

- TopBar에 토너먼트 스트립: 레벨(SB/BB/앤티) · 다음 레벨 카운트다운 · 잔존 N/48 · 내 전역 순위.
- 마일스톤 오버레이: 레벨업(기존 SnG 공지 승계), 브레이크(카운트다운), 버블 돌입(H4H 안내 —
  "다른 테이블을 기다리는 중…" 대기 표시), 파이널 진출, 테이블 이동(§4-3 연출).
- 탈락: EliminationNotice 승계(전역 순위 표기) → [순위표 보기]로 토너먼트 상세 전환.
- 우승/입상: TournamentResultOverlay 승계 + 미야코/파트너 축하 대사(dialogue situationKey
  `mtt:final`, `mtt:win` 등 — 기존 캐시·생성 파이프라인에 키만 추가).

### 5-3. 디렉터 콘솔 (운영 — 도쿠 클럽의 씨앗)

- 접근: 토너먼트 상세에 개설자(directorProfileId 일치)만 [운영] 패널 노출.
- v1 기능(P0 세트 — 리서치 §3-2): 수동 시작, **일시정지/재개**, 정지 중 블라인드 구조 수정,
  플레이어 강제 제거(환불 여부 선택), 토너 취소(환불 규칙 자동 적용), 등록자 명단.

  기존 라우트+탭+폴링 3단 패턴 복제, `getAdminTournamentSummaries()` 신규.

## 7. 테스트 전략

1. **엔진 단위**: BB 앤티(팟 회계 불변식·사이드팟·validActions), maxPlayers 가변 회귀,
   전역 순위 주입 API, mtt 모드에서 checkTournamentEnd 비활성.
2. **밸런서 단위**: 인원차 판정, 다음 BB 이동자 선정, worst position 좌석 배정(SB 금지),
   브레이크 트리거·랜덤 배정, 파이널 통합, 캐릭터 충돌 회피. (philios33 테스트 케이스 참고)
3. **매니저 통합**: 멀티엔진 하네스(setupTable 일반화판) — 등록→체크인→시작→탈락→밸런싱→이동→
   파이널→정산 풀 시나리오, H4H 배리어, 일시정지 중 블라인드 수정, 취소 환불.
4. **불변식**: 전 테이블 칩 총합 = 엔트리 × 시작 스택 (이동·탈락·브레이크 전 구간).
5. **시뮬레이션 = 부하 게이트**: 봇 48명 풀 토너먼트 자동 완주(레벨 ms 단축) — CI 통과 조건이자
   100/200명 확장 전 성능 측정 도구(핸드 처리량, emit 수, GC).
6. **소켓 통합**: table-move 계약(구 탭 격리·resync·room-lost와의 상호작용), tournament:<id> 룸.
