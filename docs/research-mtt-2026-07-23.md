### 1-2. philios33 밸런싱 알고리즘 (이식 가치 최상)

- 적정 테이블 수: `ceil(잔존 인원 / 테이블 정원)`
- 브레이크 대상 선정: `테이블 점수 = Σ(플레이어당 10 + 그 플레이어의 누적 이동 횟수)` 최소 테이블
- 좌석 배정: 테이블 크기 2~10인별 `fromPosition × toPosition` 가중치 행렬
  - `이동 점수 = (과거 이동 횟수 × 50) + 포지션 전이 점수`
  - 0점 = 완벽(SB였던 사람 → 새 테이블 D), 15점 = 최악(BB → 또 BB)
  - 이동 이력 가중치 50으로 같은 사람이 반복적으로 끌려다니는 것 방지
- 데드 버튼 인식 + **"방금 핸드 끝낸 테이블 id"를 인자로 받는 API** — 밸런싱은 각 테이블의
  핸드 종료 이벤트마다 트리거된다는 설계 힌트
- 난이도 스펙트럼: Truholdem식 "largest→smallest while diff>1" 그리디는 반나절 구현이고 소규모에 충분.
  philios33 좌석 점수화는 그 위에 얹는 공정성 레이어 → **2단계 도입이 현실적**

### 1-3. Truholdem 도메인 모델

- 상태머신: `REGISTERING → STARTING → RUNNING ⇄ (PAUSED | ON_BREAK) → FINAL_TABLE → COMPLETED` (+CANCELLED)
- `BlindLevel(level, sb, bb, ante, duration)` record + STANDARD(15분)/TURBO(8분)/DEEP(20분) 프리셋
- 리바이 3조건: ①리바이 마감 레벨 이전 ②최대 횟수 미만 ③**현재 칩 ≤ 시작 스택**. 애드온은 브레이크 중 1회
- 페이아웃: 필드 크기별 계단표 (≤6명: 65/35, ≤10명: 50/30/20, ≤18명: 5등, ≤27명: 7등…)
- 밸런싱 트리거: 탈락 처리 끝에서 자동 호출, 이동 내역을 `TournamentTablesRebalanced` 이벤트로 발행
- 파이널 통합: `총 잔존 ≤ 테이블 정원 && 테이블 2개 이상` → 인원 최다 테이블로 전원 이주

### 1-4. 빌려올 것 Top 5

1. philios33 좌석 공정성 점수 행렬 + 이동 이력 가중치
2. Truholdem 상태머신 + 도메인 이벤트 계약 (이동 내역 브로드캐스트 → 클라 이동 연출 가능)
3. "핸드 종료 이벤트 단위 밸런싱" 트리거 패턴 (기존 `handleCompletedHand` 훅과 정합)
4. 리바이/애드온 3조건 룰 + 프라이즈풀 합산식(바이인+리바이+애드온)
5. 페이아웃/블라인드 데이터 주도 구조 (htnminh WSOP CSV 시드 + 프리셋 패턴 + 반올림 합계 보정)

## 2. MTT 운영 규칙 (TDA 2024/2026 · PokerStars · GGPoker)

### 2-1. 테이블 밸런싱

- **TDA Rule 11**: 밸런싱 시 "다음 BB가 될 플레이어"를 "worst position"(곧 BB가 되는 자리)으로 이동.
  BB 2연속 납부 허용. **worst position은 절대 SB가 아니다.**
- **Robert's Rules 15/17**: 이동자 = 다음 BB 차례 플레이어 자동 선택, 빈 좌석 여럿이면 가장 빨리 BB 도는 좌석.
  **7테이블 이상 인원차 2명 이내, 6테이블 이하 1명 이내 유지.**
- **TDA Rule 10 (브레이크)**: 브레이크 테이블 출신은 어떤 좌석(SB/BB/버튼 포함)이든 받고 즉시 딜인,
  단 **SB와 버튼 사이 자리에서는 딜인 금지**.
- **2026 TDA 서밋**: 랜덤 브레이크 권장 + **브레이크 순서 비공개**(고의 스톨링 방지).
- **PokerStars 룰 2.6**: 브레이크 재배정 랜덤(BB 연속 가능), 개별 밸런싱 이동은 "블라인드 기준 비슷한 포지션".
- 온라인 공통: **이동은 핸드 종료 시점에만** (관행).

### 2-2. 블라인드 구조

- 시작 스택: 온라인 레귤러 표준 **100BB** (10,000칩). 레벨 길이: 레귤러 10~15분 / 터보 5분 / 하이퍼 3분.
- 증가율: 1.25~1.5x, 실무는 관용 수열 (50/100→75/150→100/200→150/300→200/400→…).
- **BB 앤티(=1BB, BB 좌석이 테이블 몫 일괄 납부)가 현대 표준**, 통상 레벨 3~4부터.
- GGPoker 파이널 품질 장치: Blind Rollback(평균 40BB 미만 시 레벨 되돌림), 핸드 수 기반 레벨.

### 2-3. 페이아웃

- 입상 비율: 클래식 10~15%, Stars는 2017년 이후 ~18%까지 완만화. 우승 몫 총상금 20~30%.
- Stars 45인 MT-SnG 관행: **7명 입상**. 27인: 5명(37/27/18/10/8).
- **50명 권장안(설계)**: 7명 입상(14%) — 30/21/15/11/9/7.5/6.5(%).
  참가 규모별 계단표(≤24명 4명 이하, 25~34명 5명, 35~50명 7명)가 온라인 표준 UX.
- **동시 탈락(Stars 2.2)**: 같은 핸드 복수 탈락 → 핸드 시작 칩 큰 쪽 상위(**기존 SnG `handStartChips` 계약과 동일**).
  완전 동률은 상금 균등 분할(Stars) 또는 포지션 타이브레이크(GG). H4H 중엔 전 테이블을 같은 핸드로 간주.
- **hand-for-hand**: 버블(입상 1명 전) 도달 시 전 테이블이 한 핸드씩 치고 배리어 대기. 온라인은 자동 발동·해제.

### 2-4. 리바이/리엔트리/레이트 레지

- 현대 온라인은 **리엔트리가 지배적**: 버스트 후 레이트 레지 기간 내 재참가 = 새 스택·새 랜덤 좌석·엔트리 +1.
  리바이(좌석 유지 칩 구매)·애드온(브레이크 1회)은 별도 UI·브레이크 연동이 필요해 후순위 권장.
- 레이트 레지: 시계 시간 기준(예: 시작 후 45~60분 / 레벨 6~7 종료), **입상권 도달 시 조기 마감**.
  늦게 들어와도 풀 스택, 즉시 딜인(포스팅 의무 없음). GG는 스톨링 악용 대응으로 랜덤 분산 배치.

### 2-5. 브레이크 / 파이널 / 자리비움

- 브레이크: 온라인 표준은 **동기화 브레이크(매시 55분 5분)** — 다중 토너먼트 동시 참가 대비.
  단일 토너먼트만이면 레벨 기반도 무방.
- 파이널: 잔존 ≤ 테이블 정원이면 마지막 브레이크로 자연 구성. 온라인은 리드로 생략 관행.
  GG는 '좌석 선택' 연출(칩 적은 순서로 10초씩 선택) — 캐릭터 게임 정체성과 잘 맞음.
- 자리비움/끊김(Stars 4.1): 미접속이어도 딜인+블라인드·앤티 납부, 액션 오면 체크/폴드.
  좌석·칩·순위는 종료까지 보존 — **기존 SnG away 정책과 동일 계약**.
- 레벨 전환(TDA Rule 23): **단일 토너먼트 시계, 새 레벨은 각 테이블의 "다음 핸드"부터 적용** —
  테이블 간 비동기 핸드 경계와 충돌하지 않는 핵심 원리.
- ICM 딜 메이킹: 1차 제외 권장(파이널 2~6명 전용 엣지 기능, 전원 동의 UX + ICM 계산기 공수).

## 3. 클럽 플랫폼 백오피스 조사

### 3-1. 플랫폼 한줄 요약

- **PPPoker**: 백오피스 5메뉴(이메일/멤버/칩 카운터/데이터/설정), 클럽 레벨 월 구독제, GPS/IP/기기 제한 기본 ON + 본사 100지표 자동 콜루전 감지. 정산은 100% 앱 밖(구조적 리스크).
- **PokerBros**: **매니저별 권한 개별 토글(Manager Restrictions)**이 교과서. 버블 프로텍션/파이널 딜/RIT 차별 기능.
- **ClubGG**: **MTT 개설 폼이 가장 투명** — 블라인드 프리셋 4종(Slow/Standard/Turbo/Hyper)+레벨 2~40분,
  레이트 레지 10~120분, 리엔트리 무제한 토글, 테이블 2~9인, 인원 5~10,000, 페이아웃 프리셋 3종. 클럽 칩 무료 발행.
- **Poker Now**: 무가입 MTT 운영 도구의 최소 완결 세트 — 승인 대기열·일시정지(+정지 중 블라인드 수정)·
  플레이어 제거·전자동 밸런싱·**체크인**(탑승 그룹식 노쇼 방지)·초대 코드 일괄 발급(CSV).
- **PokerStars Home Games**: 정산·보안을 플랫폼 본체가 책임지는 기준선. 클럽 리더보드/시즌이 리텐션 정석.
- **Suprema(반면교사)**: "부정행위 감시는 클럽 오너 책임" 명시 — 플랫폼 감시 부재가 신뢰 문제의 본질.

### 3-2. 운영자 백오피스 우선순위 체크리스트

**P0 (MTT 출시 필수 — 토너먼트 디렉터 최소 세트)**
- 토너 개설 폼 (ClubGG 필드 세트가 업계 표준 스펙)
- 자동 테이블 밸런싱 (전 플랫폼 공통 자동 — 수동 밸런싱 UI는 어디에도 없음)
- 일시정지/재개 + 정지 중 블라인드 구조 수정 (Poker Now 방식 — 별도 레벨 조정 기능보다 단순·동일 효과)
- 참가자 승인 대기열 / 플레이어 제거(환불 여부 선택) / 토너 취소+자동 환불
- 플레이어 측 토너 로비: 구조 미리보기·순위표·평균 스택·다음 레벨·탈락 순위 안내·테이블 이동 안내
- 핸드 정본 기록과 토너 ID 조인 (분쟁 대응 — 기존 `table_hand` 전역 핸드 ID 활용)

**P1 (도쿠 클럽 출시 필수)**
- 클럽 생성+ID+가입 승인제, 역할 2단(오너/매니저 — 권한 개별 토글), 멤버 강퇴/차단/정지
- 클럽 칩 원장(불변 기록), 클럽 대시보드(/admin의 클럽 스코프 축소판)
- 토너 취소 규칙 사전 고지(앱들이 전부 미공개라 분쟁 소지 — **명문화 자체가 차별화**)
- 감사 로그: 운영자 개입 전건 기록+사유

**P2 (차별화)**
- 체크인(단순형: 시작 N분 전 재확인, 미체크인 좌석 해제), 초대 코드 일괄 발급
- 버블 프로텍션/파이널 딜, KO 바운티, 클럽 리더보드/시즌, 즐겨찾기 구성 저장
- 콜루전 신호 리포트(같은 IP 동석 경고 — client-address 재사용, 차단보다 신호 표시가 안전)

### 3-3. 무가입·익명 세션 플랫폼의 함정 (핵심 8)

1. **신원 영속성 없이 클럽 멤버십 불성립** — 차단 멤버가 시크릿창으로 재가입하는 걸 못 막음.
   클럽 기능 전에 최소 계정(복구 코드라도)이 선행 조건.
2. **MTT는 세션 유실 비용이 캐시보다 훨씬 큼** — 토큰 유실 = 좌석·순위·상금 청구권 상실.
   → 토너 등록 시 프로필(기존 profileAuth)과 좌석 강결합, **상금은 프로필에 귀속**.
3. **멀티 엔트리 콜루전이 공짜** — 브라우저 4개 = 4엔트리 + 칩 덤핑 가능.
   같은 IP 다중 등록은 경고/신호 표시(CGNAT 오탐 학습 반영 — 차단은 위험).
4. **노쇼 극대화** — 무료 등록이라 유령 좌석 비율이 실돈 앱보다 높음. **체크인이 P2가 아니라 P0에 가까움.**
5. **운영자 권한 인증** — 오너 권한은 복구 가능한 프로필에 귀속 + 권한 변경 감사 로그.
6. **취소=자동 환불이 가능한 게 오히려 강점** — 지갑이 플랫폼 안에 있으므로(에스크로 패턴 확장).
7. **분쟁 중재 근거는 핸드 기록** — 클럽 오너에게는 자기 클럽 스코프 감사 뷰만, 홀카드 정본은 플랫폼 어드민 전용
   (오너도 잠재적 부정행위자 — 클럽 앱 생태계의 반복된 교훈).
8. **칩과 돈의 분리는 우리에겐 설계 그 자체** — 무가치 플레이 칩이라 실돈 정산 리스크 없음.
   사설 정산(칩 실돈 거래)은 약관·UI 문구로 명시적 금지.

## 4. 코드베이스 확장점 분석 (실코드 검증)

### 4-1. 최대 충돌 지점 3곳

1. **엔진 우승 판정**: `engine.ts` `checkTournamentEnd` — "한 테이블 생존 1명 = 우승 + finished=true".
   MTT에선 테이블 브레이크 트리거이지 종료가 아님.
2. **상금 풀 산정**: `startTournament`이 상금을 "이 테이블 칩 합계"로 계산. MTT는 전체 바이인 풀.
3. **1세션 1테이블 + 로비 튕김**: `session.roomId` 단일값 + `leaveAllSeatsExcept` + room-lost는 무조건
   로비행. 서버 주도 무중단 테이블 이동 계약(`table-move`)이 신규로 필요.

### 4-2. 재사용 가능한 뼈대

- **arena-runtime.ts가 상위 매니저의 청사진** — RoomManager 공개 API(createRoom/joinRoom/disposeRoom/leaveRoom)만으로
  방 오케스트레이션 가능함을 증명. 단 아레나는 1매치=1방, MTT는 1토너=N방+밸런싱.
- **`handleCompletedHand`(room-manager)가 밸런싱/브레이킹 개입 최적 지점** — scheduleNextHand 직전.
- **`pendingRemoval` + `addPlayer` 정렬 삽입 = 테이블 이동 프리미티브** — 핸드 사이 실행 시
  버튼/블라인드/팟 회계 불변식 유지. 단 이동은 칩 보존 + 경제 정산 없는 이석 경로 신규 필요
  (`leaveRoom`은 wallet 정산을 트리거하므로 그대로 못 씀).
- 엔진 좌석 로직은 배열/seatIndex 기반이라 **maxPlayers 값만 키우면 자동 동작**. 6 하드코딩은
  `RoomConfig.maxPlayers` 리터럴 타입, create-room, economy `===6`, table-layout.ts 좌표(수기 튜닝 필요).
- `recordFinish`/`assignFinishPlaces`(동시 탈락 handStartChips 계약), EliminationNotice,
  TournamentResultOverlay, SnG away 정책, 종료 후 보존 메커니즘 — 대부분 재사용 가능.
- 블라인드: `blind-schedule.ts`의 `levelIndexAt` 재사용 가능(공용 시계 공유), `SNG_PRIZE_SPLIT` 3인 고정은 일반화 필요.
- 경제: `reserveSngEntry`/`settleSngTournament` 골격 유효, `===6` 전제와 "단일 방=전체 참가자" 전제 수정 필요.
- 어드민: 라우트+탭+폴링 3단 패턴 그대로 복제 가능 (실제 탭은 8개 — AGENTS.md "7탭"은 문서 드리프트).

### 4-3. 확장성 (50명/~9테이블, 인메모리 단일 인스턴스)

기능적으로 가능. 우려 순위:
1. **봇 AI(LLM) 동시 호출** — 봇 비중 크면 병목/레이트리밋 (토너먼트 단위 대사 게이팅 필요)
2. 상태 변화당 `getPublicState` 재직렬화 비용 (테이블당 인원×액션 빈도)
3. 토너먼트 광역 공지 — Socket.io 방=룸 1:1이라 `tournament:<id>` 룸 신규 필요
- 타이머 절대 수는 낮은 우려 (수십 개 규모)

### 4-4. 테스트

- `setupTable`/`completeRunout`/칩 불변식 헬퍼 재사용 (단 setupTable 6 고정 → 가변 하네스 신규)
- 신규: 멀티엔진 하네스, 밸런싱 단위 테스트, 전역 순위·페이아웃 정산 테스트,
  봇 전용 풀 토너먼트 시뮬레이션(= 부하 테스트 겸용)

## 5. 종합 결론

1. **길은 명확하다**: TournamentManager(신규, 아레나 패턴) + 기존 PokerEngine N개 + `handleCompletedHand` 훅
   개입 + pendingRemoval/addPlayer 이동 프리미티브. 엔진의 단일 테이블 전제 2곳만 mtt 모드로 우회.
2. **규칙은 확정 가능**: TDA/Stars/GG 규칙이 상세히 문서화되어 있고 기존 SnG 계약(동시 탈락, away 정책)과
   이미 정합 — 50명 권장 파라미터표(§2-6)를 그대로 채택하면 됨.
3. **개설 폼은 ClubGG 필드 세트가 표준 스펙**, 디렉터 기능 최소 세트는 Poker Now가 증명
   (승인 대기열·일시정지·제거·자동 밸런싱·체크인).
4. **익명 세션 함정 중 MTT v1에 직결되는 것**: 상금의 프로필 귀속(§3-3 #2), 노쇼 대응 체크인(#4),
   멀티 엔트리 신호(#3). 클럽 기능은 계정 체계 선행 없이는 불가(#1).
5. **밸런싱은 2단계 도입**: v1 그리디(diff>1, 다음 BB 이동, 곧 BB 좌석 배정) → v2 philios33 공정성 점수.
