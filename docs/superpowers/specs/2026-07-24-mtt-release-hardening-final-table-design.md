# MTT 출시 하드닝 및 파이널 테이블 설계

작성일: 2026-07-24  
대상: Poker Doku 캐시 게임 공통 엔진 및 멀티테이블 토너먼트(MTT)

## 1. 목표

이번 변경은 다음 세 문제를 하나의 출시 하드닝 작업으로 해결한다.

1. 매칭되지 않은 올인 초과 베팅을 쇼다운 전에 반환해 팟·레이크·히스토리를 정확하게 만든다.
2. H4H, 브레이크, 일시정지, 테이블 병합이 충돌하지 않는 MTT 상태 모델을 만든다.
3. 마지막 한 테이블의 첫 핸드 전에 서버 권위 파이널 인트로와 벚꽃 챔피언십 테마를 제공한다.

단순 시각 효과만 추가하지 않는다. QA에서 실제 칩, 순위, 상금 또는 게임 진행 중단에 영향을 주는
문제는 같은 배포 차단 범위로 다룬다.

## 2. 이번 범위

### 포함

- 캐시·SnG·MTT 공통 매칭되지 않은 베팅 반환
- wallet 캐시 레이크와 핸드 히스토리 정합성
- 공개 `lastAction` 금액 정규화
- 홀수 칩의 버튼 왼쪽 우선 지급
- MTT 합성 가능한 보류 사유
- H4H 진입 전 테이블 밸런싱 및 파이널 병합
- 파이널 테이블 형성/인트로/플레이 상태
- 서버 deadline 기반 파이널 인트로, 브레이크, 일시정지 표시
- 동일 핸드 동시 탈락 규칙과 공동 상금
- wallet MTT 참가자 방장의 진행 중 경제 조작 차단
- 정지 중 레벨 변경의 다음 핸드 적용
- 로비에 있는 자리비움 참가자의 강제 화면 이동 방지
- MTT 문구, 상태 배너, 모바일 정보 우선순위
- 빈 토너먼트 TTL과 개설자별 제한
- A/B/C 파이널 테마 프리셋

### 제외

- 늦은 등록, 리엔트리, 리바이
- 바운티, 새틀라이트
- ICM 딜 협상
- 파이널 좌석 선택, 블라인드 롤백, 체스 클록
- 여러 토너먼트 동시 플레이
- 완전한 관전자 모드

## 3. 매칭되지 않은 베팅

### 문제

현재 엔진은 큰 스택의 올인 전액을 `totalContributed`에 남기고 단독 자격 사이드팟을 만든다.
그 결과 런아웃 중 큰 스택은 0칩으로 보이고, wallet 캐시에서는 상대가 매칭하지 않은 금액에도
레이크가 부과된다. 쇼다운에서 그 단독 팟을 다시 이기는 방식은 회계·표시·히스토리 모두 잘못이다.

### 규칙

베팅 라운드가 닫히는 순간, 해당 스트리트의 `currentBet` 중 유일한 최고 금액이 두 번째로 높은
매칭 가능 금액을 넘으면 그 차이를 반환한다. 이전 스트리트에 이미 매칭된 기여금은 다시 비교하지
않는다.

- 반환 전에 앤티인 `deadContributed`는 비교에서 제외한다.
- 반환액만큼 `chips`를 증가시킨다.
- 같은 금액만큼 `currentBet`과 `totalContributed`를 감소시킨다.
- `state.currentBet`은 남은 최고 `currentBet`으로 다시 계산한다.
- 팟은 반환 이후의 기여금으로 `rebuildPots()`한다.
- 반환은 `uncalled-return` 히스토리 액션으로 남긴다.
- `lastAction`은 플레이어가 요청한 원본이 아니라 실제 투입량으로 정규화한다.
- 반환된 금액은 팟, 레이크, 승리액, 수익 계산에 들어가지 않는다.
- 정산 입력에는 자격자가 한 명뿐인 팟이 존재하면 안 된다.

반환은 `isBettingRoundComplete()`가 참이 된 뒤 `advanceStreet()` 또는 폴드 승리 정산에 들어가기
직전에 한 번 수행한다. 이 시점이면 더 이상 그 베팅에 응수할 플레이어가 없어 반환액이 확정된다.

## 4. MTT 상태 모델

### 공개 단계

`TournamentState.stage`를 다음 값으로 확장한다.

```ts
type TournamentStage =
  | 'multi-table'
  | 'final-forming'
  | 'final-intro'
  | 'final-playing'
  | 'complete';
```

`TournamentState`에는 다음 공개 정보도 포함한다.

```ts
interface TournamentState {
  stage?: TournamentStage;
  holdReasons?: TournamentHoldReason[];
  stageEndsAt?: number;
  finalTheme?: FinalTableTheme;
}
```

`stageEndsAt`은 `final-intro`와 시간 기반 브레이크에 사용하는 epoch ms다. 클라이언트는 로컬에서
추측한 시간을 상태 전이 근거로 사용하지 않고 표시용 카운트다운에만 사용한다.

### 내부 보류 사유

테이블별 단일 `held: Map<roomId, reason>`을 합성 가능한 `Map<roomId, Set<reason>>`으로 바꾼다.

```ts
type TournamentHoldReason =
  | 'director-pause'
  | 'scheduled-break'
  | 'h4h-barrier'
  | 'final-forming'
  | 'final-intro';
```

테이블은 보류 사유 집합이 비었을 때만 다음 핸드를 시작할 수 있다. 한 사유를 해제해도 다른 사유가
남아 있으면 `resumeRoom()`을 호출하지 않는다.

우선순위는 다음과 같다.

1. 완료/취소
2. 운영자 일시정지
3. 예정 브레이크
4. 파이널 테이블 병합과 인트로
5. H4H 배리어

### 파이널 전이

```text
MULTI_TABLE
  → FINAL_FORMING
  → FINAL_INTRO
  → FINAL_PLAYING
  → COMPLETE
```

- 생존자가 테이블 정원 이하이고 테이블이 둘 이상이면 모든 유휴 테이블에 `final-forming`을 건다.
- 진행 중 핸드는 끝까지 진행한다.
- 모든 테이블이 유휴가 되면 한 테이블로 전원을 이동한다.
- 모든 이동과 좌석 정렬이 끝날 때까지 목적지 방을 재개하지 않는다.
- 파이널 방에 `final-intro`를 걸고 `stageEndsAt = now + 4_500`을 주입한다.
- deadline이 지나면 `final-intro`만 해제하고 첫 핸드를 시작한다.
- `final-playing`에 진입한 뒤에는 재접속이나 테이블 스냅샷 재수신으로 인트로를 반복하지 않는다.
- 시작부터 한 테이블인 MTT도 첫 핸드 전에 같은 인트로를 한 번 거친다.

### H4H 순서

핸드 완료 처리는 다음 순서를 지킨다.

1. 탈락 후보 수집
2. 진행 중인 동일 H4H 라운드가 있으면 전 테이블 완료 배리어
3. 해당 라운드 탈락 순위 확정
4. 완료 여부 확인
5. 파이널 테이블 조건 확인 및 병합
6. 테이블 밸런싱/브레이크
7. 다음 H4H 라운드 무장
8. 남은 보류 사유가 없는 방만 재개

H4H permit은 실제 `startHand()`가 성공해 `handNumber`가 증가한 뒤 소비한다. 시작이 실패하면
permit을 유지해 테이블이 영구 보류되지 않게 한다.

## 5. 동시 탈락과 상금

### 정렬 규칙

동일 핸드에서 여러 명이 탈락하면:

1. `handStartChips`가 큰 플레이어가 상위다.
2. 같은 테이블에서 시작 스택까지 같으면 버튼 왼쪽에서 먼저 만나는 좌석이 상위다.
3. H4H의 서로 다른 테이블에서 시작 스택까지 같으면 공동 순위 그룹이다.

서버 이벤트 도착 시간이나 0.01초 단위 탈락 시각은 사용하지 않는다. 네트워크·이벤트 루프·테이블
완료 순서가 순위에 영향을 주기 때문이다.

### 공동 상금

공동 순위 그룹이 차지하는 순위 구간의 상금을 합산하고 인원수로 정수 균등 분배한다.
나머지 1칩 단위는 결정론적으로 `playerId` 오름차순에 한 칩씩 배분한다. 이는 포커 플레이 우위를
뜻하지 않고 DB에 정수 상금을 저장하기 위한 회계 결정 규칙이다.

표시 결과는 같은 `place`를 허용한다. 다음 순위는 그룹 크기만큼 건너뛴다. 예를 들어 공동 5위 두 명
다음은 7위다. 경제 저장소는 입력 배열 길이와 총상금 보존을 검증하되 순위의 유일성은 요구하지 않는다.

## 6. 운영 안전장치

### Wallet MTT 권한

- 시작 전: 방장은 등록 관리와 시작/취소를 할 수 있다.
- 시작 후: 참가자 방장의 `remove-player`, `set-level`, `cancel`을 거절한다.
- practice MTT: 기존 디렉터 기능을 유지한다.
- wallet의 예외 운영 개입은 현재 소켓 방장 권한으로 노출하지 않는다.

### 레벨 변경

정지 중 `set-level`은 토너먼트 시계와 `pendingLevelIndex`만 바꾼다. 진행 중 핸드의
`smallBlind`, `bigBlind`, `ante`, `minRaise`는 바꾸지 않는다. 각 테이블의 다음 `startHand()`
직전에 pending 레벨을 적용한다.

### 이동 이벤트

서버는 플레이어 세션의 현재 `roomId`가 이동 출발 방과 같을 때만 `table-move`를 보낸다.
로비에서 자리비움 중이면 서버 좌석과 복귀 목적지만 갱신한다. 클라이언트도
`fromRoomId === currentRoomId`일 때만 화면을 교체한다.

### 생성 남용

- 등록자 0명인 토너먼트는 짧은 TTL 후 자동 취소한다.
- 프로필당 동시에 소유할 수 있는 등록 중 토너먼트 수를 제한한다.
- 자동 취소는 기존 `cancel`/환불/방 정리 경로를 재사용한다.

## 7. 파이널 테이블 UX

### 인트로

첫 핸드 전에 전체 테이블을 덮는 인트로를 약 4.5초 표시한다.

- `FINAL TABLE`
- 남은 참가자 수
- 1위 상금 또는 총상금
- 토너먼트 이름
- 인트로 중 액션 버튼과 턴 타이머 비활성
- 재접속 시 남은 시간만 표시
- `prefers-reduced-motion`에서는 확대와 입자 이동 제거

### 테마 프리셋

```ts
type FinalTableTheme =
  | 'sakura-championship'
  | 'gold-spotlight'
  | 'neon-arena';
```

- A `sakura-championship`: 기본 활성. 따뜻한 분홍/금색, 밝은 펠트, 벚꽃 입자.
- B `gold-spotlight`: 금색 스포트라이트, 어두운 펠트, 클래식 메이저 대회 분위기.
- C `neon-arena`: 청록/마젠타 조명, 높은 대비, e스포츠 아레나 분위기.

세 프리셋은 같은 DOM과 상태 흐름을 사용하고 CSS 변수와 효과 레이어만 바꾼다. 이번 배포의 생성
화면에는 선택기를 노출하지 않고 A를 기본값으로 저장한다. 운영 설정에서 값을 바꾸면 B/C를 바로
사용할 수 있다.

### 상태와 문구

테이블 및 상세 화면은 다음 상태를 표시한다.

- 핸드 포 핸드 진행 중
- 테이블 재편성 중
- 브레이크
- 운영자 일시정지
- 파이널 테이블

MTT 탈락 안내에서 `Sit & Go`와 실제로 제공하지 않는 `관전` 표현을 제거한다. 프리즈아웃,
늦은 등록/리엔트리 없음, 체크인 조건을 상세 화면에 명시한다. 모바일 TopBar는 레벨, 블라인드,
남은 인원, 현재 보류 상태를 우선하고 부가 정보는 상세 모달로 보낸다.

## 8. 검증 전략

필요한 계약만 회귀 테스트한다.

1. 프리플랍과 포스트플랍의 매칭되지 않은 베팅 반환
2. 반환 후 팟, 레이크, 공개 액션, 핸드 히스토리
3. 홀수 칩의 버튼 왼쪽 지급
4. 1명 대 4명 버블에서 추가 핸드 없이 파이널 병합
5. H4H와 브레이크/일시정지/파이널 보류 사유 조합
6. 동일 스택 탈락의 같은 테이블 타이브레이크와 교차 테이블 공동 상금
7. wallet MTT 진행 중 방장 권한 차단
8. 진행 중 핸드의 레벨 불변과 다음 핸드 적용
9. 자리비움 참가자 이동과 정상 접속 참가자 이동
10. 파이널 인트로 최초 1회, deadline, 첫 핸드 차단
11. 빈 토너먼트 TTL과 개설 제한

UI 전용 테스트를 대량 추가하지 않는다. 타입 검사, 린트, 프로덕션 빌드, 데스크톱·모바일 실제
브라우저 시나리오로 테마, 인트로, 상태 배너, 입력 차단을 검증한다.
